Architecture Approaches

How should we structure Duolingo-for-Claude under the hood?

A

Single-Page App (SPA)

One HTML file. All 8 modules, lessons, and game logic live in a single page. JavaScript handles routing, transitions, and state. Feels like an app.

Pros

  • Instant navigation, no page reloads
  • Smooth Duolingo-like transitions and animations
  • One file to deploy

Cons

  • Gets complex fast with 8 modules x many lessons
  • All content loads upfront (can get heavy)
  • Harder to maintain and extend
B

Multi-Page Static + Shared Shell

Each module gets its own HTML file. A shared CSS/JS shell handles the Duolingo chrome (sidebar, XP bar, progress). Lessons defined as JSON data files.

Pros

  • Clean separation: content (JSON) vs presentation
  • Easy to add lessons — edit a JSON file
  • Each page loads only what it needs

Cons

  • Page transitions not as seamless
  • More files to manage
C

SPA + JSON Curriculum (Recommended)

Single-page app shell for the Duolingo experience. All curriculum content lives in JSON data files (one per module). App loads modules on demand.

Pros

  • App-like feel with smooth transitions
  • Content separated from code — add lessons by editing JSON
  • Lazy-loads modules (fast initial load)
  • progress.json sits alongside curriculum data
  • Scales to product without rewriting

Cons

  • Slightly more upfront JS architecture
